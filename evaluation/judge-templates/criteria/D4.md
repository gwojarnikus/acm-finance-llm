# Criterion D4: Internal Quantitative Consistency

## Definition

Verification of arithmetic correctness. Do the components sum to the totals claimed? Do derived ratios follow from the underlying data?

This criterion addresses the well-documented LLM weakness in arithmetic reasoning (Tonmoy et al., 2024). LLMs can generate plausible-looking numbers that don't actually compute correctly. D4 forces explicit calculation verification.

## CRITICAL: Tool use is MANDATORY

You MUST use the Python interpreter or calculator to verify every arithmetic claim. Do NOT "eyeball" numbers. If you do not have access to a Python interpreter, set `score: null` and `skipped_reason: "tool_use_unavailable_for_D4"`.

This is not optional. Without explicit computation, your D4 score is unreliable, and the Soft Veto threshold (2.0) could be falsely triggered.

## Soft Veto threshold lowered to 2.0

The Soft Veto threshold for D4 is **2.0** (not 3.0 as in the draft). Rationale:

- LLM arithmetic errors are well-documented (Tonmoy et al., 2024; Cemri et al., 2025).
- Without tool use, judges may falsely score low on D4 due to their own arithmetic errors.
- Lowering the threshold to 2.0 means λ = 0.5 is activated only when there are obvious contradictions (not minor rounding issues).

This calibration choice will be re-validated after Stage 3b pilot (12 reports). If cross-model agreement on D4 (Claude vs GPT-4o in Stage 6) is low (ICC < 0.6) or false-positive rate exceeds 20%, threshold will be lowered further OR D4 prompt refined in v2 (e.g., more constrained tool-use protocol, structured calculation template). Human expert evaluation is NOT a fallback option in this workflow (decision 2026-05-17).

## N/A rule (from draft §3.5)

If the report contains NO numerical data (which would be unusual for financial analysis), apply the N/A rule:
- Set `score: null`
- `skipped_reason: "D4_NA_no_numerical_data"`
- The penalty for "no numerical data when expected" is borne by D1 (Operational Actionability) and A1 (Data Grounding), not D4.

This avoids "double penalization" for the same defect (empty set has no contradictions, by mathematical definition).

## Behavioral anchors (from BARS pilot v1)

- **Level 0:** Obvious arithmetic contradiction (e.g., sum of components ≠ stated total; ratio doesn't match underlying data).
- **Level 2 (poor):** Multiple minor inconsistencies (e.g., rounding errors propagating through calculations).
- **Level 5 (average):** Most calculations correct, 1–2 inconsistencies.
- **Level 8 (good):** All calculations consistent; components sum correctly. Observed in R1 (e.g., S1-R1-G line 25: PIT/CIT table 28.8 + 174.1 = 202.8 ✓, 63.7 + 28.1 = 91.8 ✓, 92.5 + 202.2 = 294.6 [actually 294.7, 0.1 rounding] — very close consistency).
- **Level 10 (excellent):** Plus explicit derivations with verifiable steps (formulas + intermediate steps). Observed in R1 (e.g., S1-R1-G lines 53–55: `Required flat rate τ = T / B = 92.5 / 8341.5 = 1.11%` — explicit formula with parameters).

## Scoring method

**Type:** arithmetic check with mandatory tool use.

**Procedure:**

1. **Numerical claim inventory.** Extract every numerical claim from the report:
   - Sums (e.g., "total revenue PLN X = component A + component B")
   - Ratios (e.g., "margin = profit / revenue = X%")
   - Products / scalars (e.g., "tax = rate × base = X")
   - Derived metrics (e.g., "Sharpe = (return - risk-free) / volatility")
2. **For each claim, execute the calculation:**
   ```python
   # Example: report claims "total = 28.8 + 174.1 = 202.8"
   actual = 28.8 + 174.1
   print(actual, actual == 202.8)  # expect 202.9, mismatch by 0.1 (rounding)
   ```
3. **Classify discrepancies:**
   - **Rounding (≤0.5% of value):** acceptable, no deduction
   - **Minor (0.5–5% of value):** -0.5 per instance
   - **Major (>5% or wrong direction):** -2 per instance
4. **Bonus for explicit derivation:** +1 if report includes formulas + intermediate steps + units.
5. Apply N/A rule if no numerical data present.

Score guidance:
- All calculations correct + explicit derivations → 9–10
- All calculations correct (rounding tolerated) → 7–8
- Most calculations correct, 1–2 minor errors → 5–6
- Several inconsistencies → 3–4
- Obvious arithmetic contradiction → 0–2 (Soft Veto trigger if < 2.0)
